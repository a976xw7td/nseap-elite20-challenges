**核心挑战：** 不是做一个学术 demo，而是**在真实企业环境中落地**——数据是真实的、需求是业务驱动的、交付标准是客户定义的。

**适合谁：** 有技术背景、对知识图谱/本体/agent 有兴趣、能写代码的人。

### Proposal 必须包含的 5 个部分

---

#### Part 1：项目选择（Project Selection）

```markdown
## 1. 项目选择

- 我选择的项目：（A / B / C）
- 为什么适合我：
  - 我的相关经验：____
  - 我在 C1–C7 中展示的相关能力：____

```

不要泛泛而谈"我很感兴趣"。用你在 C1–C7 中的**具体产出**来证明你适合。

---

#### Part 2：拿来主义分析（Reference + Adaptation）

> **这是最重要的评分项。**

```markdown
## 2. 拿来主义分析

### 参考来源
| 来源 | 链接 | 核心思想 |
|------|------|----------|
| （项目/公司/开源方案 1） | | |
| （项目/公司/开源方案 2） | | |
| （项目/公司/开源方案 3） | | |

### 拿来的部分
- 我准备直接复用：____
- 为什么好用：____

### 修改 / 删除 / 重构的部分
- 我准备去掉：____（原因：____）
- 我准备重构：____（原因：____）
- 我准备新增：____（原因：____）
```

**评分标准：**
- 列了 0 个来源 → 不合格（说明没做调研）
- 列了来源但没有取舍分析 → 及格
- 列了来源 + 明确的 adopted/removed/improved → 良好
- 列了多个来源 + 对比分析 + 创造性组合 → 优秀

---

#### Part 3：执行方案（Execution Plan）

用 **KSTAR 结构**撰写：

```markdown
## 3. 执行方案（KSTAR）

### K（Knowledge）：我已有的知识和资源
- 相关技能：____
- 可用工具：____
- 团队 / 协作资源：____

### S（Situation）：当前问题是什么
- 现状：____
- 痛点：____
- 约束条件：____

### T（Task）：我要解决什么
- 核心目标：____
- 子任务分解：
  1. ____（Week 1-2）
  2. ____（Week 3-4）
  3. ____（Week 5-6）
  4. ____（Week 7-8）

### Â（Action Plan）：怎么做
- 每个子任务的执行步骤
- 使用什么 AI / Agent / 工具
- 每一步的输入和输出

### R̂（Expected Result）：预计产出什么
- 可交付物清单：
  1. ____
  2. ____
  3. ____
- 成功标准：____
- 验证方法：____
```

**注意 R̂ 是"预期结果"**——入选后每周 AAR 会对比 R̂ vs R（实际结果），这个差值就是你的学习信号。预期越具体，AAR 越有价值。

---

#### Part 4：你的角色（Your Role）

你不是"参与者"。你必须定义自己是什么：

```markdown
## 4. 我的角色

- 角色定义：（builder / architect / operator / researcher）
- 我负责的具体模块：____
- 我的边界（我不负责什么）：____
- 我需要的支持：____
```

| 角色 | 含义 |
|------|------|
| **Builder** | 动手做——写代码、搭系统、产出可交付物 |
| **Architect** | 设计整体结构——定义系统架构、数据模型、流程 |
| **Operator** | 运营执行——部署、运维、客户对接、数据收集 |
| **Researcher** | 调研分析——技术选型、竞品分析、方案评估 |

一个人可以兼多个角色，但必须有一个主角色。边界越清晰，执行越高效。

## 五、入选后的要求

被选中的人，身份正式转变为**项目执行者（Operator）**。以下是强制要求：

### 5.1 每周真实产出（Weekly Deliverable）

每周必须交付至少一项**可验证的产出**：

| 类型 | 示例 |
|------|------|
| 可运行系统 | 部署的服务、可测试的 API、可使用的工具 |
| 文档 | 技术文档、客户交付材料、方案书 |
| Agent Workflow | 可执行的 agent 配置 / skill / pipeline |
| 数据 | 本体模型、知识图谱、标注数据、测试结果 |

**"在做了"不算产出。"做出来了"才算。** 每个产出必须是别人能看到、能验证、能使用的东西。

### 5.2 强制 AAR（After Action Review）

每周必须提交 AAR 报告：`姓名_C8_AAR_W周次.md`

```markdown
## Weekly AAR — Week N

### 预期（R̂）
本周计划产出：____

### 实际（R）
本周实际产出：____

### 差值（ΔR = R̂ − R）
- 完成了什么：____
- 没完成什么：____
- 意外发现：____

### 失败分析
- 哪里失败了：____
- 根本原因：____
- 不是根本原因但看起来像的：____

### 下周计划
- 基于本周 AAR，下周的 R̂ 调整为：____
- 需要改变的做法：____
- 需要的支持：____
```

**AAR 的核心不是"汇报进度"，而是暴露 ΔR（预期与实际的差值）。** ΔR 越大说明你的预测能力需要校准；ΔR 持续缩小说明你在进步。这正是 KSTAR 循环的学习信号。

### 5.3 拿来主义持续执行

入选后不是"方案定了就执行"。你必须持续：

- **找更好的方案** — 每周至少评估一个新的参考来源
- **替换已有方案** — 发现更好的就换，不要沉没成本
- **优化系统** — 每个迭代都要有可衡量的改进

每周 AAR 中必须包含一条：**"本周我发现了____，它比我之前的方案更好，因为____。"**

找不到更好方案也要写："本周我评估了 X 和 Y，但现有方案仍然是最优的，因为____。" 这说明你在持续检索和判断。

## 七、淘汰机制

以下情况会被**移出项目**：

| 行为 | 为什么不行 |
|------|-----------|
| ❌ 只讨论，不做 | 讨论不产生结果。结果产生结果。 |
| ❌ 只做 demo，没有真实应用 | C6 是做 demo 的地方。C8 是做真事的地方。 |
| ❌ 连续两周不提交 AAR | 没有 AAR = 没有学习 = 没有进步。 |
| ❌ 不做拿来主义（闭门造车） | 连续两周 AAR 中没有新的参考来源 = 停止学习。 |
| ❌ 产出无法被验证 | "我做了"但拿不出可验证的东西 = 没做。 |

**淘汰不是惩罚——是保护项目和团队。** 真实项目有真实的 deadline 和客户。不能执行的人留在项目里，会拖累能执行的人。
