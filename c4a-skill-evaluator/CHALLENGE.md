# Challenge C4A：技能提交自动评审（Automated Skill Submission Evaluator）

> 🔥 C4 的进阶挑战——不是**做**一个技能，而是**评审**所有人的技能。
> 本质：把 wechat-doc-mapper 的"分类盘点"能力升级为"自动评审"能力。

---

## 背景

C4（技能分享与传播）要求每位同学提交一个可复用、可执行、可验证、IO明确的技能包。大家把文件发到 WeChat 群里——但问题来了：

- 谁交了？谁没交？
- 交的内容**齐不齐**？（五个必须文件：skill说明、可执行内容、demo、教学说明、AI日志）
- 交的内容**合不合格**？（四条件：可复用、可执行、可验证、IO明确）
- 能不能**自动给出初步评审**，而不是等老师一个个看？

现有的 `wechat-doc-mapper` 技能已经能做第一步（扫描→分类→缺口分析），但它只是"收发室"——能分信，不能阅卷。

**C4A 的任务：把收发室升级为自动阅卷系统。**

---

## 与 C4 的关系

| | C4 | C4A |
|--|----|----|
| 你做什么 | 创建一个技能包，教别人用 | 创建一个**评审技能包**的技能 |
| 输入 | 你的能力 + AI | WeChat 群文件夹里所有人的 C4 提交 |
| 输出 | 可安装的 .skill 包 | 自动化评审报告（完整性检查 + 质量评分 + 改进建议） |
| 本质 | 能力产品化 | **元能力产品化**（评审能力本身变成技能） |
| 与 wechat-doc-mapper 的关系 | 无直接关系 | 以它为起点，升级改造 |

完成 C4A 的同学同时满足 C4 的要求——因为 C4A 的产出本身就是一个可复用、可执行、可验证的技能。

---

## 挑战目标

构建一个 **C4 提交自动评审技能**，实现以下完整流水线：

```
WeChat 群文件夹
       ↓
  ① 文件采集与识别
       ↓
  ② 提交完整性检查
       ↓
  ③ 技能质量评审（四条件）
       ↓
  ④ 评审报告生成
```

---

## 挑战结构

| 阶段 | 时间 | 性质 | 产出 |
|------|------|------|------|
| **C4A 方案设计** | 发布 → +3天 | 设计阶段 | 方案文档（怎么做、架构图、评审维度定义） |
| **C4A 实现提交** | +3天 → +7天 | 开发阶段 | 可安装运行的技能包 + 真实评审报告 |

**🏆 首个提交可运行且正确评审至少 3 份真实 C4 提交的同学，直接获得 C4 最高评级。**

---

## 适用原则

**⚡ AI-First 原则：** 整个技能的开发过程必须 AI 驱动。你可以用 Claude Code / Cursor / ChatGPT 写代码，用 AI 设计评审维度，用 AI 生成报告模板。每个手动步骤必须反向举证。

**📚 拿来主义原则：** 必须从 `wechat-doc-mapper.skill` 出发。不奖励从零开始——奖励"拿来 → 理解 → 改造 → 超越"。说明你拿了什么、改了什么、为什么改。

**📛 命名规范：** 所有提交文件遵循 `姓名拼音_C4A_内容描述.扩展名` 格式。

---

## 四级任务（由浅入深）

你不需要全部完成。每一级都是独立可提交的成果，越深分越高。

### Level 1：文件采集器（Bronze 🥉）

> **从 WeChat 群文件夹中识别所有 C4 相关提交，按作者分组。**

输入：一个本地文件夹路径（WeChat 同步文件夹、手动下载文件夹、或任何包含群文件的目录）
输出：按作者分组的 C4 文件清单

核心能力：
- 扫描文件夹，识别文件名中的 `_C4_` 标记
- 按命名规范提取作者名
- 处理不规范命名（回退到文件内容/元数据识别）
- 支持混合格式（.md, .docx, .pdf, .skill, .py, .mp4, .png）

**这一级本质上就是 wechat-doc-mapper 的 C4 聚焦版。** 如果你理解了 wechat-doc-mapper 的代码，这一级可以在 30 分钟内完成。

---

### Level 2：完整性检查器（Silver 🥈）

> **对每个作者的提交，检查 C4 要求的五个必须文件是否齐全。**

在 Level 1 基础上，增加：

| 必须文件 | 检测信号 |
|----------|----------|
| Skill 说明文档 | 文件名含 `skill说明` / `skill_doc` / `skill-doc`，或 .md 文件内容含"输入""输出""使用场景" |
| 可执行内容 | `.skill` / `.py` / 含 YAML frontmatter 的 .md / 有代码块的文档 |
| Demo | `.mp4` / `.mov` / `.png` / `.jpg` / `.gif` / 含 `demo` 的文件名 |
| 教学说明 | 文件名含 `教学` / `tutorial` / `teaching`，或内容含"上手""常见坑""步骤" |
| AI 日志 | 文件名含 `AI日志` / `AI_log`，或内容含"使用的 AI 工具""prompt""迭代次数" |

输出：每个作者的完整性报告（✅ 齐全 / ⚠️ 部分缺失 / ❌ 严重缺失）+ 具体缺什么

---

### Level 3：质量评审器（Gold 🥇）

> **对每个提交的技能内容，按 C4 四条件进行自动评审。**

这是核心创新部分。你需要设计**如何用 AI（或规则+AI）自动评估**以下四个维度：

| 条件 | 自动评审思路参考 |
|------|-----------------|
| **可复用（Reusable）** | 说明文档是否包含环境要求？是否有安装步骤？是否依赖特定本地路径？是否有泛化说明？ |
| **可执行（Executable）** | 是否包含可运行代码/prompt/workflow？代码是否有语法错误？.skill 包结构是否完整？ |
| **可验证（Verifiable）** | 是否定义了明确的输入输出？是否有测试用例或 demo？是否有预期结果？ |
| **IO 明确（Clear I/O）** | 说明文档中是否有"输入____，输出____"的明确描述？类型是否清晰？ |

每个维度给出：评级（✅ / ⚠️ / ❌）+ 评审依据 + 改进建议

**关键设计决策（你需要自己选择）：**

| 方案 | 优点 | 缺点 |
|------|------|------|
| 纯规则（关键词/正则） | 确定性强，速度快 | 覆盖率低，容易误判 |
| 纯 LLM | 理解力强，灵活 | 不确定，慢，贵 |
| 规则 + LLM 混合 | 规则快筛 + LLM 深审 | 架构复杂 |

你的方案选择本身就是评审的加分项——在方案文档中说明你为什么选这个方案。

---

### Level 4：完整评审系统（Platinum 💎）

> **生成完整的班级评审报告，含排名、改进建议、趋势追踪。**

在 Level 3 基础上，增加：

- **班级总览仪表板**：全班提交状态一览（谁交了、谁没交、总体质量分布）
- **排名系统**：基于四条件评分的综合排名（可配置权重）
- **改进建议生成**：针对每个作者，生成个性化的"下一步行动"
- **版本追踪**：检测 `_v2`、`_v3` 等迭代版本，展示进步轨迹
- **输出格式**：Markdown 报告 + Excel 详表 + （可选）HTML 可视化

## WeChat 文件获取（开放题）

> ⚠️ **文件怎么从 WeChat 群下载到本地？这一步留给你自己解决。**

可能的路径（仅提示方向，不限于此）：

| 方案 | 思路 |
|------|------|
| 手动下载 | 最简单——群文件逐个保存到文件夹。可行但不自动化 |
| WeChat 桌面端同步 | 利用 WeChat PC/Mac 端的文件自动同步功能，找到本地缓存路径 |
| 群文件导出工具 | 调研是否有现成工具（拿来主义！） |
| 自建方案 | 如果你有技术能力，可以探索更自动化的获取方式 |

**关键约束：** 无论你怎么获取文件，你的技能必须接受**一个本地文件夹路径**作为输入。这样无论文件怎么来的，技能都能工作。

如果你在文件获取这一步有创新方案（比如自动化同步脚本），这是额外加分项。

## 评审标准

| 维度 | 权重 | 说明 |
|------|------|------|
| **评审准确性** | 30% | 你的技能给出的评审结果是否合理？对真实提交的评审是否准确？ |
| **完成级别** | 25% | Level 1–4 完成到哪一级？ |
| **拿来主义质量** | 20% | 是否充分理解并改造了 wechat-doc-mapper？取舍分析是否清晰？ |
| **可复用性** | 15% | 别的老师 / 助教能直接用你的技能评审 C4 提交吗？ |
| **创新性** | 10% | 评审维度设计、文件获取方案、报告格式等有没有独到之处？ |

## 起步指南

### Step 1：理解现有技能（Day 1）

```
安装 wechat-doc-mapper.skill → 跑一遍 → 读 SKILL.md → 读 wechat_doc_mapper.py → 读 challenges.yaml
```

问自己：
- 它怎么识别作者的？
- 它怎么把文件映射到挑战的？
- 它的缺口分析逻辑是什么？
- 如果我要加"质量评审"，应该在哪个环节插入？

### Step 2：设计评审维度（Day 1–2）

重新读 C4.md，把四条件翻译成**可检测的信号**：

```
"可复用" → 有安装说明？不含绝对路径？有环境要求说明？
"可执行" → 有代码块？有 YAML frontmatter？文件可被解析？
"可验证" → 有测试用例？有示例输入输出？有预期结果？
"IO 明确" → 文本中出现"输入"和"输出"？有一句话描述？
```

### Step 3：改造 + 开发（Day 2–5）

以 wechat-doc-mapper 为基座，增加：
1. C4 专用的完整性检查逻辑
2. 内容读取 + 评审逻辑（规则 / LLM / 混合）
3. 报告生成模板

### Step 4：真实测试（Day 5–7）

用群里已有的 C4 提交测试你的技能。把评审结果发到群里请大家验证——这同时也是你的 C4 传播。

### Step 5：提交 + 迭代

按命名规范提交到群里。根据反馈迭代 v2、v3。

## 为什么 C4A 值得做

```
C4  = 我会做一个技能
C4A = 我会评审所有人的技能
```

这是从"手艺人"到"质量官"的跃迁。

在真实世界里，能做事的人很多，能**系统化地评估**做事质量的人更稀缺。C4A 训练的是：

1. **元认知**——理解"什么是好技能"比"做一个技能"更深一层
2. **系统设计**——把模糊的评审标准变成可执行的自动化流程
3. **拿来主义实战**——在已有工具上改造，而不是从零造轮子

完成 C4A 后，你拥有的不只是一个技能——而是一个**可用于任何技能评审场景的自动化框架**。

---

> 📣 **C4A 的终极价值：**
>
> C4 是把能力变成产品。C4A 是把**评审能力**变成产品。
> 一个能自动评审技能的技能，本身就是最好的技能。
> 这就是递归的力量——也是 Agent 系统最核心的设计模式。
